# 内容生产引擎：开发者视角产品介绍与功能说明

版本：v0.3 internal
适用项目：content-engine
运行地址：http://127.0.0.1:3000/
核心目标：把「关键词规划 + 站点知识库 + 品牌 SOP + 模型生成 + QA 质检 + 发布资料包」整合成一个可复用的文章工厂框架。

---

## 1. 产品定位

内容生产引擎不是一个单纯的文章生成器，而是一个面向多站点运营的内容生产后台。它的设计目标是让运营人员用中文维护配置、SOP、知识库和内容策略，同时让模型按目标站点语言输出符合品牌规范、SEO 结构、QA 规则和发布要求的英文文章。

当前系统支持从外部数据包导入站点配置、作者档案、品类知识、内链库、关键词任务、运行模板、样式参考和项目级写作指令。导入完成后，运营人员可以在 Web 后台检查配置、生成大纲、生成文章、查看 QA、复制发布包，并下载 HTML 或整站输出 ZIP。

适合的使用场景：

- 单个独立站持续生产 SEO 博客内容。
- 多个商品站复用同一套文章工厂框架。
- 把 Claude Project / 人工 SOP 转换为可维护的数据结构。
- 将关键词表、知识库、品牌约束、内链策略和 QA 标准统一沉淀。
- 在真实使用中根据效果不断迭代 Prompt、模板和质检逻辑。

## 2. 技术结构总览

项目入口是 `main.js`。它启动 Express 服务，托管 `ui/` 静态文件，并把所有后端能力挂载到 `/api`。

核心目录：

```text
content-engine/
├── main.js                    # Express 启动入口
├── engine/
│   ├── api.js                 # 后端 API、导入、站点管理、生成流程入口
│   ├── config-loader.js        # 加载站点配置、知识库、模板、Project 指令
│   ├── keyword-store.js        # 关键词 CSV 读写和任务状态管理
│   ├── task-queue.js           # 批量任务队列和并发控制
│   ├── prompt-builder.js       # 大纲 Prompt 和文章 Prompt 构建
│   ├── generator.js            # 模型调用与文章生成
│   ├── splitter.js             # 生成结果拆分
│   ├── cleaner.js              # HTML 清洗
│   ├── qa.js                   # 质检评分、硬性拦截、警告
│   ├── meta-writer.js          # meta-table.xlsx 写入
│   └── link-matcher.js         # 内链匹配
├── ui/
│   └── index.html              # 单页后台 UI
├── templates/
│   ├── article-types.json      # 全局文章类型策略
│   ├── prompt-sections.json    # 全局 Prompt 区块
│   └── qa-rules.json           # 全局 QA 规则
└── sites/
    └── [siteId]/
        ├── site.json
        ├── author.json
        ├── knowledge.json
        ├── links.json
        ├── style-reference.json
        ├── project-instructions.md
        ├── keywords.csv
        ├── templates/          # 站点级模板覆盖，可选
        ├── outlines/           # 大纲 JSON
        ├── outputs/            # HTML、QA、数据包、元数据表
        └── docs/               # 导入报告、迁移记录、参考资料
```

设计原则：

- `engine/` 是执行逻辑层，负责稳定生产。
- `sites/[siteId]/` 是站点数据层，真实业务数据按站点隔离。
- `templates/` 是全局默认策略层，站点可用自己的 `sites/[siteId]/templates/` 覆盖。
- `ui/index.html` 是运营后台层，不引入复杂前端框架，便于快速维护。

## 3. Web 界面总览

后台是单页应用，顶部固定导航，左侧为全局运行控制区，右侧根据面板切换展示不同功能。

### 3.1 顶栏

顶栏高度 46px，承担全局入口。

功能元素：

- Logo：`内容生产引擎`。
- 站点下拉：切换当前 `siteId`，例如 Zerevia、Nookcraft。
- 主面板标签：关键词任务、站点配置、运行逻辑、模型设置、上传文件管理。
- 站点管理：位于站点下拉旁，打开弹窗创建新站点；创建成功后自动切换到新站点并进入上传文件管理。
- API Key 输入框：运行时传给生成接口，不写入项目文件。
- 导入数据文件/ZIP：进入通用导入流程。
- 导出 CSV：下载当前站点关键词表。
- 下载文章 ZIP：打包下载当前站点 `outputs/`。
- 元数据表格：下载 `meta-table.xlsx`。

切换站点时，前端会重新加载关键词、配置、模板，并清空已选状态，关闭抽屉。

## 4. 左侧运行控制区

左栏固定宽度 280px，所有面板共用。

### 4.1 统计数字

四个统计块：

- 总计：关键词任务总数。
- 已完成：状态为 `done` 的文章数量。
- 待生成：未生成或排队中的任务数量。
- 失败：状态为 `failed` 的任务数量。

这些数字来自 `/api/sites/:siteId/keywords` 返回的关键词行状态。

### 4.2 运行控制

按钮包括：

- 运行已选文章生成：只对当前勾选的关键词执行文章生成。它会调用 `generate-article`，要求该关键词已经有大纲；如果要一键全量生成，先点击表头复选框全选。
- 运行已选大纲：只对当前勾选的关键词批量生成大纲，用于先审大纲再生成正文。
- 重试失败：重新执行失败任务。
- 重置全部：把全部任务状态改回待生成。
- 重置已选：只重置勾选行。
- 并发数滑块：控制前端批量请求并发数。

当前文章生产逻辑是明确两阶段：先生成大纲，再根据大纲生成文章。单行操作中可以分别执行「纲」和「生成」；批量操作以“已选”为核心，通过表头全选保留一键全量处理能力。这样可以避免误触全站生成，也更适合先抽查大纲、再批量生成正文的运营流程。

### 4.3 运行日志

运行日志位于左侧底部，最新日志在最上方。用途：

- 展示导入结果。
- 展示保存配置成功或失败。
- 展示大纲生成、文章生成状态。
- 展示 API 错误、QA 失败、文件下载等反馈。

产品原则：普通 API 错误不弹窗，而是进入日志，避免打断运营流程。

## 5. 面板一：关键词任务

关键词任务是日常生产内容的主工作台。

### 5.1 关键词表格

表格字段：

- 复选框：用于批量重置或未来批量操作。
- 关键词：核心关键词，可内联编辑。
- URL Slug：文章输出文件名和 URL slug，可内联编辑。
- 优先级：P1/P2/P3。
- 文章类型：购买指南、价格指南、对比评测、FAQ 问答、操作教程、支柱文章等。
- 目标字数：用于 Prompt 和 QA 参考。
- 状态：待生成、生成中、大纲完成、已完成、失败。
- QA 分：文章生成后的综合评分。
- 操作：预览、生成大纲、生成文章、删除。

关键词表来自 `sites/[siteId]/keywords.csv`，由 `engine/keyword-store.js` 读写。

### 5.2 新增关键词

点击「新增关键词」展开表单。

字段包括：

- 关键词。
- URL Slug，留空可自动生成。
- 优先级。
- 文章类型。
- 目标字数。
- 搜索意图。
- 次要关键词。
- 变体/长尾词。
- 内链 URL。
- 文章方向/备注。

保存后调用 `POST /api/sites/:siteId/keywords`，写入 `keywords.csv` 并进入待生成状态。

### 5.3 文章抽屉

点击预览按钮后，从右侧滑出抽屉。抽屉是生产结果的集中查看区。

抽屉标签页：

- 大纲预览：读取 `outputs/[slug].outline.json` 或 `outlines/[slug].json`。
- 文章预览：渲染 `outputs/[slug].html`，并结合 `style-reference.json` 做基础样式预览。
- HTML 源码：显示原始 HTML。
- QA 检查：读取 QA 结果，显示综合得分、硬性拦截、警告和各项评分。
- 发布包：展示 SEO 标题、Meta Description、ALT 文案、Schema JSON-LD、发布前检查清单。
- 数据包：展示大纲、QA、发布包、生成 Prompt、关键词行等结构化数据。

下载单篇文章时，应使用 `/api/sites/:siteId/articles/:slug?download=1`，避免浏览器直接打开 HTML 源码页面。

## 6. 面板二：站点配置

站点配置面板负责管理某个站点的业务数据和品牌规则。顶部显示当前站点，并提供「保存所有更改」和「删除当前站点」。删除站点有二次确认，避免误删。

### 6.1 品牌设置：site.json

对应文件：`sites/[siteId]/site.json`

字段包括：

- 网站 ID。
- 网站名称。
- 域名。
- 语言。
- 网站定位。
- 目标受众。
- 品牌角色。
- 转化目标。
- 写作语气。
- 禁止规避的写作风格。
- 必须表达。
- 禁止表达。

后端读取后会进入 `ctx.site`，被 Prompt 构建和 QA 规则使用。

### 6.2 作者档案：author.json

对应文件：`sites/[siteId]/author.json`

字段包括：

- 作者姓名。
- 职位/身份。
- 背景介绍。
- 写作风格说明。
- 故事素材库。

故事素材库支持 JSON 数组结构，适合存放 Claude Project 中的一手经验素材、编号故事、真实客户问题、创始人经历等。此前发现的 `[object Object]` 显示问题已经修复：UI 现在会保留 JSON 结构，不再把对象数组错误转成字符串。

生成正文时，故事素材库会被作为“已批准的一手经验来源”传入 Prompt。模型应从故事库中选择与当前主题最相关的细节，改写成可直接发布的自然段落，而不是输出【需要一手经验】、TODO、E-E-A-T marker 等编辑占位符。如果故事库里没有完全匹配的素材，模型应写成更克制的事实/观察表达，不能留下待补标记。

### 6.3 品类知识：knowledge.json

对应文件：`sites/[siteId]/knowledge.json`

核心区块：

- 术语表：中英文术语和释义。
- 权威事实：可验证事实、行业标准、材料参数。
- 买家高频问题：FAQ 和回答。

这些数据用于 Prompt 中的事实约束，防止模型凭空发挥。

### 6.4 内链库：links.json

对应文件：`sites/[siteId]/links.json`

核心区块：

- 专题页：最高优先级。
- 已发布博客：用于同主题互链和关键词蚕食判断。
- 产品分类页：用于转化导向 CTA 和产品线支撑。

`engine/link-matcher.js` 会根据关键词和文章类型匹配合适内链。

### 6.5 样式参考：style-reference.json

对应文件：`sites/[siteId]/style-reference.json`

用途：保存目标网站文章 HTML 的排版和组件风格参考。

当前 UI 已改为一个大输入框 `styleBrief`，运营人员可以直接用自然语言填写样式规则，而不是拆成很多字段。结构化旧字段仍会保留，但生成时优先读取 `styleBrief`。

建议写清楚：

- 文章根节点 class，例如 `<article class="zerevia-article">`。
- 标题、段落、列表、表格的阅读节奏。
- FAQ 是否使用 `<details><summary>...</summary></details>` 手风琴。
- 引用块、经验块、提示块、警告块、CTA、Related links 的 HTML 结构和 class。
- 颜色和字体气质，例如 deep purple、lavender accent、warm ivory、editorial calm 等。
- 禁止输出 full-page HTML、`<style>`、`<script>`、`<head>`、`<body>`、nav、footer。

它不直接替代目标站主题 CSS，但会强约束模型输出带 class 的 CMS 可嵌入 HTML。最终视觉效果仍取决于目标站是否有对应 CSS；如果目标站没有这些 class 的样式，需要在主题或文章模板中补 CSS。

### 6.6 Project 指令：project-instructions.md

对应文件：`sites/[siteId]/project-instructions.md`

这是专门为 Claude Project 这类长规则设计的模块。建议存放：

- 你是谁。
- 品牌世界观。
- 写作禁区。
- 合规边界。
- E-E-A-T 规则。
- 文章结构规则。
- 发布检查清单。
- 标准输入/输出格式。

优先级建议：

1. 结构化配置文件优先，例如 site.json、author.json、knowledge.json。
2. Project 指令作为整体写作框架。
3. 单篇关键词任务中的备注和方向作为最具体约束。

如果三者冲突，越具体、越接近当前文章任务的信息优先。

## 7. 面板三：运行逻辑

运行逻辑面板用于调整生成策略和 QA 标准。

### 7.1 文章类型策略：article-types.json

用途：定义不同文章类型的生成边界，但不保存固定标题句式或固定开头模板。

当前 UI 已改为单个 JSON 填写框，而不是拆成多张字段卡片。这样做的原因是：文章类型策略本身应该服从全局 Project 指令和单篇关键词任务，如果在 UI 中硬编码 `titleFormat` 或 `openingTemplates`，会导致文章套模板、同质化，并且容易与站点 SOP 冲突。

推荐保留字段：

- 类型 ID：例如 `buying-guide`、`faq`、`how-to`。
- `wordRange`：默认字数范围，关键词行的 `targetwordcount` 优先级更高。
- `schemaType`：默认 Schema 类型，例如 `Article`、`FAQPage`、`HowTo`。
- `strategy`：文章类型原则，用来描述读者意图、结构重点、证据要求和转化角色。
- `structureNotes` / `avoid`：可选，用来说明结构注意事项和应避免的写法。

不建议保留字段：

- `titleFormat`：容易让标题同质化。
- `openingTemplates`：容易让开头句重复，尤其会和 Project 指令中的品牌写法冲突。

当前前端保存时会自动移除 `titleFormat` 和 `openingTemplates`。全局 `templates/article-types.json` 也已改为通用 fallback 策略，不包含 Nookcraft、Zerevia 或其它具体站点的硬编码句式。站点差异应放在 `sites/[siteId]/templates/article-types.json`、`project-instructions.md`、知识库和关键词任务中。

### 7.2 Prompt 区块：prompt-sections.json

用途：维护通用 Prompt 片段。

典型区块：

- 作者声明格式。
- 观察性结论写法。
- 内链规则说明。
- CTA 风格要求。

这些区块会被 `prompt-builder.js` 组合进大纲 Prompt 和文章 Prompt。

### 7.3 QA 规则：qa-rules.json

用途：定义硬性拦截、警告和评分权重。

QA 分层：

- Hard Fail：必须拦截，例如出现禁用词、缺少作者模块、没有核心关键词等。
- Warning：需要人工复核，例如没有产品页内链、存在关键词蚕食风险、缺少一手经验等。
- Score Weights：SEO 结构、品牌匹配、内容有用性、可读性、发布就绪度。

当前 QA 已支持硬性拦截和警告分开显示。分数不是唯一判断标准：84 分也可能因为 Hard Fail 而不可发布。

## 8. 面板四：模型设置

当前支持：

- Xiaomi MiMo V2.5 Pro。

## 9. 面板五：上传文件管理与站点管理

当前版本把“新增站点”和“上传站点资料”拆开：

- 新增站点属于左上角「站点管理」弹窗，不再作为主导航中的站点级页面。
- 上传文件管理是当前已选站点的数据入口，只处理当前站点的品牌文档、作者档案、知识库、内链表、关键词表、样式参考和 Project 指令。

这样做的原因是：新增站点是全局管理动作，而上传文件是某个具体站点的数据维护动作。二者分开后，运营更容易理解“当前正在写入哪个站点”，也能降低不同站点变量数据串写的风险。

站点管理弹窗用于创建一整套站点目录。

输入字段：

- 网站 ID。
- 网站名称。
- 域名。
- 语言。
- 网站定位。
- 作者姓名。
- 作者职位。
- 写作语气。
- 目标受众。
- 必须表达。
- 禁止表达。

提交后调用：`POST /api/sites/create`

后端会创建：

- `site.json`
- `knowledge.json`
- `author.json`
- `links.json`
- `style-reference.json`
- `project-instructions.md`
- `keywords.csv`
- `outputs/`
- `outlines/`

创建成功后，前端刷新站点下拉，自动切换到新站点，并跳到站点配置面板。

## 10. 通用导入数据文件/ZIP

导入入口是当前版本的重要模块。它解决的问题是：实际站点数据通常不止一个 CSV，而是一整包配置文件、模板、报告和关键词。

### 10.1 前端流程

1. 用户点击「导入数据文件/ZIP」。
2. 选择多个文件，或直接选择 `.zip`。
3. 前端读取文件内容，`.xlsx` 和 `.zip` 使用 base64。
4. 调用 `/api/sites/:siteId/import-preview`。
5. 后端解析 ZIP，识别每个文件的目标位置和导入状态。
6. 前端展示待导入文件列表。
7. 用户勾选需要导入的文件。
8. 调用 `/api/sites/:siteId/import-files` 正式写入。

### 10.2 可识别文件

核心配置：

- `site.json` → `sites/[siteId]/site.json`
- `author.json` → `sites/[siteId]/author.json`
- `knowledge.json` → `sites/[siteId]/knowledge.json`
- `links.json` → `sites/[siteId]/links.json`
- `style-reference.json` → `sites/[siteId]/style-reference.json`
- `project-instructions.md` → `sites/[siteId]/project-instructions.md`

关键词：

- `keywords.csv` → 合并导入 `keywords.csv`
- `.xlsx` 关键词表 → 转成 CSV 后合并导入

模板：

- `article-types.json` → `sites/[siteId]/templates/article-types.json`
- `prompt-sections.json` → `sites/[siteId]/templates/prompt-sections.json`
- `qa-rules.json` → `sites/[siteId]/templates/qa-rules.json`

生成资料：

- `outline.json` 或 `*.outline.json` → `outlines/[slug].json`

### 10.3 重复识别

- 新文件：目标不存在，可导入。

## 12. 执行链路：从关键词到文章

### 12.1 加载上下文

`config-loader.js` 读取站点目录，组装上下文 `ctx`：

- `ctx.site`
- `ctx.author`
- `ctx.knowledge`
- `ctx.linkIndex`
- `ctx.styleReference`
- `ctx.projectInstructions`
- `ctx.templates`
- `ctx.outputDir`

### 12.2 构建大纲 Prompt

`prompt-builder.js` 使用：

- 当前关键词行。
- 文章类型策略。
- 品牌设置。
- 作者档案。
- 知识库。
- 内链库。
- Project 指令。
- Prompt 区块。

输出大纲生成所需的 systemPrompt 和 userPrompt。

### 12.3 调用模型生成大纲

`api.js` 中的 `generateOutline()` 调用模型接口。模型必须返回可解析 JSON。后端会做 normalize，确保至少有：

- slug
- keyword
- articleType
- workingTitle
- sections
- faq
- qaRisks

大纲写入：

- `outputs/[slug].outline.json`
- `outlines/[slug].json`

### 12.4 基于大纲生成文章

`generateArticleFromOutline()` 会读取大纲，再调用 `generator.js` 生成文章 HTML。

输出结果包括：

- 原始生成内容。
- 清洗后的 HTML。
- SEO meta。
- QA 结果。
- data-pack。

写入文件：

- `outputs/[slug].html`
- `outputs/[slug].qa.json`
- `outputs/[slug].data-pack.json`
- `outputs/[slug]-data-pack.json`
- `outputs/meta-table.xlsx`

### 12.5 QA 检查

`qa.js` 对清洗后的 HTML、关键词行和上下文执行检查。

输出：

- 综合得分。
- 分项得分。
- hardFails。
- warnings。
- status：pass / review / blocked。

重要逻辑：

- 有 hardFail 时，不能仅因为分数高就视为通过。
- warning 不一定阻止发布，但应进入人工复核。
- 站点级 QA 规则可以覆盖通用规则。

### 12.6 元数据输出

`meta-writer.js` 写入 `meta-table.xlsx`，包括：

- Sheet1：文章记录。
- Sheet2：SEO 标题和 Meta Description。
- Sheet3：ALT 文案。
- Sheet4：Schema。

发布包接口会合并 HTML、meta-table 和 data-pack 中的信息，方便运营直接复制发布。

## 13. 数据优先级和冲突处理建议

当前建议的优先级如下：

1. 单篇关键词任务字段：最具体，例如 article direction、internal links、target word count。
2. 结构化站点配置：site.json、author.json、knowledge.json、links.json、style-reference.json。
3. 站点级 Project 指令：project-instructions.md。
4. 站点级模板：sites/[siteId]/templates/*.json。
5. 全局模板：templates/*.json。
6. 引擎内置默认值。

冲突示例：

- 如果 Project 指令说“每篇 1200-1800 词”，但关键词行写了 `targetwordcount=900`，应以关键词行为准。
- 如果 Project 指令中有内链规则，但 links.json 中没有对应专题页，应由 QA 给出 warning，而不是模型编造 URL。
- 如果中文 SOP 中写了英文输出要求，应该保留中文配置，但 Prompt 明确 `Write reader-facing content in English`。

## 14. 当前已知优化方向

### 14.0 站点管理与上传文件管理

当前 UI 将站点初始化拆成两个阶段：

- 第一阶段在左上角「站点管理」弹窗填写基础信息：网站 ID、网站名、域名、语言、定位、作者、语气、受众和表达禁区，用于创建站点目录和默认文件骨架。
- 第二阶段进入主导航「上传文件管理」页上传网站变量数据：支持 `.json`、`.csv`、`.md`、`.txt`、`.xlsx` 和 `.zip`。ZIP 会在后端解压后逐个文件识别。

上传文件管理会展示当前站点的必备文件清单：

- `site.json`：品牌/网站变量。
- `author.json`：作者档案和故事素材库。
- `knowledge.json`：品类知识库。
- `links.json`：内链库。
- `style-reference.json`：文章 HTML 样式参考。
- `project-instructions.md`：站点级 Project 指令。
- `keywords.csv`：关键词规划表。

后端新增接口：

- `POST /api/sites/:siteId/setup-preview`：接收上传文件，先用现有导入规则做启发式识别；如果请求中带有 API Key，则调用模型判断文件属于哪个模块、应写入哪个目标文件、置信度是多少，并返回可勾选的识别列表。
- `POST /api/sites/:siteId/setup-apply`：只写入用户勾选的识别结果。JSON 配置会与现有配置做递归合并，关键词表走去重导入，模板文件写入站点级 `templates/`，无关文件会被标记为 `ignore`。

### 14.1 实际使用反馈修复记录

## 15. 给老板汇报时可以这样概括

这套系统的价值不是“让 AI 写文章”，而是把过去散落在人工 SOP、Claude Project、关键词表、知识库、内链表和发布检查清单里的内容生产流程，沉淀成一个可重复运行的后台。

它带来的效率提升主要体现在：

- 站点复制：新站点只需要导入配置包和关键词表，就能复用生产框架。
- 标准一致：品牌语气、禁用词、合规规则、内链规则和 QA 标准集中管理。
- 过程可见：从关键词、大纲、文章、QA 到发布包，每一步都有界面和文件记录。
- 质量可控：不是直接让模型输出成品，而是先大纲、再正文、再 QA、再发布包。
- 人机协作：中文母语运营可以维护中文配置，模型按要求输出英文内容。
- 可持续迭代：每次真实生产中的问题都可以沉淀回模板、Project 指令或 QA 规则。

最终目标是从“人工逐篇写博客”升级为“运营维护内容系统，系统批量生产可审核文章”的模式。




### 14.3 图片配置与插图插入模块

系统在文章抽屉中新增“图片配置”标签页，用于把文章从纯文本 HTML 推进到可发布的图文稿：

- 后端接口 `GET /api/sites/:siteId/image-plan/:slug` 会根据已生成文章的 HTML、关键词行、H2 结构和 `style-reference.json` 自动生成图片规划。
- 默认规则：1 张封面图；正文每 1200 词推荐 2 张插图，不足 1200 词仍建议至少 2 张正文插图；当前实现最多生成 8 张正文插图，避免长文一次性过载。
- 每个图片位包含推荐场景、AI 生图提示词、ALT 文本、caption、推荐插入位置和上传状态。
- 运营可在抽屉中复制提示词去外部生图工具生成图片，也可手动修改提示词和 ALT。
- 图片上传支持两种入口：点击“选择图片”按钮，或把图片文件直接拖放到对应图片位的拖放区域。两种方式都走同一个上传接口和状态回写逻辑。
- `POST /api/sites/:siteId/image-plan/:slug/upload` 会把上传图片保存到 `outputs/images/[slug]/`，并回写 `outputs/[slug].images.json`。
- `POST /api/sites/:siteId/image-plan/:slug/insert` 会把已上传图片按图片位插入到 `outputs/[slug].html`：封面默认插到 article 开头，正文图优先插入对应 H2 后方。
- 插入的图片使用 `<figure class="content-engine-image ..." data-ce-image-slot="...">` 包裹，重复执行插入时会先移除同 slot 的旧 figure，避免重复堆叠。

当前版本的图片模块先解决“规划、上传、ALT、插入”闭环。后续如果要一键发布到 WordPress 媒体库，需要在 WordPress 发布接口中继续补充 media upload，把本地 `/api/sites/.../article-images/...` 地址替换为 WordPress attachment URL。


图片理解模型建议：如果要让系统自动识别已上传图片并生成更准确的 ALT、caption 或图片审核结果，优先接入 Google Gemini 的视觉理解接口。Google 官方文档说明 Gemini 模型支持图片理解、图片描述、分类和视觉问答，并支持 inline image 或 File API 两种传图方式。结合当前 AI Studio 免费额度，建议默认使用 `gemini-3.1-flash-lite` 做图片理解任务，因为它在当前项目额度中具备更高的 RPD/RPM；`gemini-2.5-flash` 可作为质量兜底但每日免费次数更少。Gemma 3/4 也具备图文输入能力，但更适合作为自部署开源模型路线，不适合当前本地 Node 后台直接调用 Google API 的低成本方案。

这个设计的目的，是让中文母语运营可以低成本检查文章质量，同时保证底层发布内容仍然是英文。

## 2026-05-18 上传资料站点隔离修复

本次修复解决“上传后的数据会在不同网站之间串掉”的问题。问题根因在于前端上传识别结果 `setupFiles` 原本只是一个全局临时状态，没有记录它属于哪个站点；如果用户在识别完成前后切换站点，或旧站点的异步请求晚于新站点返回，就可能把 A 站资料展示或写入到 B 站。

修复点：

- 前端新增 `setupFilesSite`，上传识别结果会绑定到发起上传时的站点。
- 切换站点时清空关键词选择、抽屉、配置缓存、上传识别结果和站点绑定，避免残留上一个站点的数据。
- 前端新增 `siteLoadSeq` 请求序号，站点切换后，旧站点晚返回的关键词、配置、建站资料状态请求会被丢弃，不再覆盖当前站点界面。
- “写入已勾选资料”会先检查识别结果所属站点是否等于当前站点，不一致时拒绝写入，并提示重新上传识别。
- `setup-preview`、`setup-apply`、`import-files` 三个后端接口新增 `siteId` 一致性校验：body 中的 `siteId` 必须和 URL 路由中的 `:siteId` 一致，否则返回 `409 Site mismatch`。

验证结果：

- 临时创建 A/B 两个测试站点。
- A 站上传识别关键词文件后，尝试把该识别结果写入 B 站，后端返回 `409`，写入被拒绝。
- 正确写入 A 站后，A 站关键词数量为 1，B 站关键词数量仍为 0。

这意味着后续即使用户在上传识别、站点切换、写入资料之间快速操作，也不会再把某个站点的资料误写到另一个站点。

## 2026-05-18 WordPress 批量发布

在关键词任务左侧运行控制区新增「发布已选到 WordPress」按钮。使用者先勾选文章，再点击批量发布；系统只发布状态为 `done/已完成` 的文章，未完成、失败或未生成的行会自动跳过并写入运行日志。

执行逻辑：
1. 前端读取当前站点 `site.json` 中的 `wordpress.endpoint / username / appPassword / defaultStatus / categories / tags`。
2. 对每个已选且已完成的关键词行，先读取 `/api/sites/:siteId/publish-pack/:slug` 获取 SEO Title 与 Meta Description。
3. 逐篇调用现有单篇接口 `/api/sites/:siteId/publish-wordpress/:slug`，沿用同一套 WordPress REST API 发布逻辑。
4. 左侧进度条显示批量发布进度，运行日志逐篇记录成功链接或失败原因。

这样保留了单篇抽屉发布能力，也补齐了批量生产完成后的“勾选后一键发布草稿”工作流。

### WordPress 批量发布条件调整

批量发布不再要求关键词行状态必须为 `done/已完成`。实际发布条件改为：该 slug 对应的 HTML 文件存在即可发布。这样当文章内容已经生成但 QA 或状态标记为失败时，运营仍可人工判断后勾选发布。

执行时，系统会逐篇检查 `/api/sites/:siteId/articles/:slug` 是否存在；没有 HTML 的文章会跳过并写入日志，存在 HTML 的文章会继续调用 WordPress 发布接口。

### WordPress 发布失败排查与重试机制

Beadlo 批量发布时出现“前几篇成功、后续 fetch failed”的情况，文章 HTML 和发布包本身存在，失败点在程序连接 WordPress REST API 阶段。为降低 WordPress 主机限流、DNS 短暂失败、连接中断造成的批量失败，发布逻辑做了两层增强：

- 后端 `publishToWordPress` 改为通过 `fetchWordPressWithRetry` 请求 WordPress，每篇最多重试 3 次；遇到 429、5xx、网络连接失败或超时会等待后重试。
- 前端批量发布每篇之间增加 1.2 秒间隔，避免连续快速创建草稿触发主机安全策略或 REST API 限流。
- 错误信息会包含更多连接细节，例如 HTTP 状态、DNS/连接错误码或超时信息，方便判断是账号权限、站点接口还是主机限制。

这不改变单篇发布接口，也不影响 Zerevia 已经可发布的流程。

### Excel 上传兼容性修复：非标准 OOXML 工作簿

`kitminiaturehouse_GSC_new_blog_keyword_plan_90.xlsx` 无法上传的原因是：文件虽然是 `.xlsx`，但内部 XML 使用 `x:` 命名空间前缀，并且关系文件里的属性顺序与 `exceljs` 预期不一致，导致 `exceljs` 抛出 `Cannot read properties of undefined (reading 'sheets')`。此前程序因此无法读取 Sheet，只能把文件判定为二进制不可读文件，目标为 `ignore`，前端复选框被禁用。

修复方式：后端新增原生 OOXML 兜底解析器。当 `exceljs` 读取失败时，程序直接读取 `xl/workbook.xml`、`xl/_rels/workbook.xml.rels`、`xl/worksheets/sheet*.xml`，解析带命名空间前缀的行列数据，并根据 `主关键词 / 目标URL / 搜索量 / KD / Blog ID / H1建议 / 内容角度 / 内链目标` 等字段转换为 `keywords.csv`。

验证结果：使用原始文件 `E:\Google download\kitminiaturehouse_GSC_new_blog_keyword_plan_90.xlsx` 测试，上传预览结果为 `keywords.csv`，状态 `ready`，识别 `90 new, 0 duplicate`。

### 关键词表文章类型映射修复与重复行更新

`kitminiaturehouse_GSC_new_blog_keyword_plan_90.xlsx` 中的 `Schema` 字段包含 `Article + FAQ`。此前导入逻辑优先匹配 `FAQ`，导致这类商业/指南型文章被误写为 `faq`。现在文章类型归一化顺序已调整为：`how-to / comparison / price-guide / pillar / article|guide|buying / faq`，因此 `Article + FAQ` 会进入 `buying-guide`，真正的问答型关键词才进入 `faq`。

导入逻辑也同步调整：再次导入同一关键词表时，不会新增重复行，但会用新表里的非空字段更新已存在的关键词记录。这样运营重新上传修正后的规划表，可以批量修正 `articletype / direction / volume / kd / blogid` 等字段，而不会制造重复关键词。

验证结果：对 `kitminiaturehouse` 重新应用同一份 90 行关键词表，结果为 `added: 0, skipped: 90, updated: 89`。前几行已从 `faq` 修正为 `buying-guide`，`how to make a diorama` 保持为 `how-to`。

本轮把 B2B 文章生成后的交付资料从普通 SEO 发布包扩展为更适合外贸站使用的 B2B 发布包，核心目标是让文章不仅能看，还能直接服务询盘、工程沟通和采购决策。

- `/api/sites/:siteId/publish-pack/:slug` 对 B2B 站点额外返回 `b2bPublish` 和 `publishCtas`。
- `b2bPublish` 会聚合关键词表中的 `buyerrole`、`funnelstage`、`industry`、`application`、`productline`、`geotarget`、`ctatype`、`evidenceneeded`、`casetarget`、`solutiontarget`、`rfqtarget`、`compliancerisk` 等字段。
- `publishCtas` 会根据站点内链库和关键词字段推荐 `request quote`、`send drawing/spec`、`download catalog`、`technical consultation`、`view solution` 等转化动作。
- B2B schema 会自动补足 `Organization`、`BreadcrumbList`，并在页面中存在产品或规格信息时补充 `Product`。CTA 会写入 `WebPage.potentialAction`，方便发布包保留页面转化意图。
- `meta-table.xlsx` 新增 B2B 发布辅助字段，并新增 `B2B Publish` 工作表。老表在重新写入时会自动追加缺失列，避免旧文件结构导致新字段丢失。
- 发布包 UI 新增 `B2B 发布信息` 和 `推荐转化 CTA` 区块，运营在文章工作台即可看到买家角色、漏斗阶段、行业、应用、证据需求和推荐转化入口。

### 站点管理与默认策略分组优化

本轮把“站点类型”从新增站点基础表单里的下拉选项，升级为站点管理弹窗的第一层分流入口。这样创建站点时先选择运营模型，再进入对应表单，避免 C端和 B端字段挤在一起。

- `站点管理` 弹窗新增两个创建入口：`创建 C端 / DTC 内容站`、`创建 B端 / 外贸 / 制造站`。
- C端创建表单保留品牌定位、目标受众、作者姓名、作者职位、写作语气、必须表达和禁止表达。
- B端创建表单改为公司/网站名称、商业模式、转化目标、公司定位、目标市场、目标行业、买家角色、核心产品、核心能力、认证/资质、MOQ、交期和禁止编造声明。
- 创建接口仍复用 `/api/sites/create`，前端根据入口自动传入 `siteType: b2c` 或 `siteType: b2b`，后端继续复用已有 B2C/B2B 默认文件生成逻辑。

默认策略也同步改为按站点类型分组管理：

- 策略编辑器新增作用域切换：`C端 / DTC 默认策略`、`B端 / 外贸 / 制造站默认策略`、`通用基础策略`。
- `/api/templates/:file` 支持 `?type=b2c`、`?type=b2b`、`?type=common`，分别读写 `templates/b2c/`、`templates/b2b/`、`templates/common/`。

### 关键词覆盖计划优化

本轮补强了关键词规划表中 `secondarykeywords`、`variants`、长尾词字段在大纲、正文和 QA 中的使用链路，避免模型只围绕主关键词写而忽略子关键词。

- 大纲生成 Prompt 新增 `keywordCoveragePlan` 要求：主关键词、子关键词、变体/长尾词都需要规划到具体位置。
- 大纲 JSON 结构新增 `keywordCoveragePlan.primary / secondary / variants`，每项包含 `term`、`plannedPlacement`、`sectionHeading` 和 `required`。
- 后端 `normalizeOutline()` 会自动补齐关键词覆盖计划；即使模型漏掉该字段，也会根据关键词表中的主关键词、子关键词和变体词生成兜底计划。
- 正文生成 Prompt 会明确要求按照大纲中的 `keywordCoveragePlan` 自然覆盖关键词，不做关键词堆砌，也不能静默忽略已提供的子关键词。
- QA 新增 `keywordCoverage` 报告，统计主关键词、子关键词、变体/长尾词的覆盖情况。
- QA 会在必需关键词缺失时追加 `keyword_coverage_missing` warning，列出缺失词，方便人工复查或重新生成。
- 文章工作台的大纲预览新增“关键词覆盖计划”表格。
- QA 检查页新增“关键词覆盖检查”表格，显示每个词是已覆盖、缺失，还是可选未用。

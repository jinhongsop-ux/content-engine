# 内容生产引擎：开发者视角产品介绍与功能说明

版本：v0.3 internal
适用项目：content-engine
运行地址：http://127.0.0.1:3000/
核心目标：把「关键词规划 + 站点知识库 + 品牌 SOP + 模型生成 + QA 质检 + 发布资料包」整合成一个可复用的文章工厂框架。

---

## 1. 产品定位

内容生产引擎不是一个单纯的文章生成器，而是一个面向多站点运营的内容生产后台。它的设计目标是让运营人员用中文维护配置、SOP、知识库和内容策略，同时让模型按目标站点语言输出符合品牌规范、SEO 结构、QA 规则和发布要求的英文文章。

当前系统支持从外部数据包导入站点配置、作者档案、品类知识、内链库、关键词任务、运行模板、样式参考和项目级写作指令。导入完成后，运营人员可以在 Web 后台检查配置、生成大纲、生成文章、查看 QA、复制发布包，并下载 HTML 或整站输出 ZIP。

适合的使用场景：

- 单个独立站持续生产 SEO 博客内容。
- 多个商品站复用同一套文章工厂框架。
- 把 Claude Project / 人工 SOP 转换为可维护的数据结构。
- 将关键词表、知识库、品牌约束、内链策略和 QA 标准统一沉淀。
- 在真实使用中根据效果不断迭代 Prompt、模板和质检逻辑。

## 2. 技术结构总览

项目入口是 `main.js`。它启动 Express 服务，托管 `ui/` 静态文件，并把所有后端能力挂载到 `/api`。

核心目录：

```text
content-engine/
├── main.js                    # Express 启动入口
├── engine/
│   ├── api.js                 # 后端 API、导入、站点管理、生成流程入口
│   ├── config-loader.js        # 加载站点配置、知识库、模板、Project 指令
│   ├── keyword-store.js        # 关键词 CSV 读写和任务状态管理
│   ├── task-queue.js           # 批量任务队列和并发控制
│   ├── prompt-builder.js       # 大纲 Prompt 和文章 Prompt 构建
│   ├── generator.js            # 模型调用与文章生成
│   ├── splitter.js             # 生成结果拆分
│   ├── cleaner.js              # HTML 清洗
│   ├── qa.js                   # 质检评分、硬性拦截、警告
│   ├── meta-writer.js          # meta-table.xlsx 写入
│   └── link-matcher.js         # 内链匹配
├── ui/
│   └── index.html              # 单页后台 UI
├── templates/
│   ├── article-types.json      # 全局文章类型策略
│   ├── prompt-sections.json    # 全局 Prompt 区块
│   └── qa-rules.json           # 全局 QA 规则
└── sites/
    └── [siteId]/
        ├── site.json
        ├── author.json
        ├── knowledge.json
        ├── links.json
        ├── style-reference.json
        ├── project-instructions.md
        ├── keywords.csv
        ├── templates/          # 站点级模板覆盖，可选
        ├── outlines/           # 大纲 JSON
        ├── outputs/            # HTML、QA、数据包、元数据表
        └── docs/               # 导入报告、迁移记录、参考资料
```

设计原则：

- `engine/` 是执行逻辑层，负责稳定生产。
- `sites/[siteId]/` 是站点数据层，真实业务数据按站点隔离。
- `templates/` 是全局默认策略层，站点可用自己的 `sites/[siteId]/templates/` 覆盖。
- `ui/index.html` 是运营后台层，不引入复杂前端框架，便于快速维护。

## 3. Web 界面总览

后台是单页应用，顶部固定导航，左侧为全局运行控制区，右侧根据面板切换展示不同功能。

### 3.1 顶栏

顶栏高度 46px，承担全局入口。

功能元素：

- Logo：`内容生产引擎`。
- 站点下拉：切换当前 `siteId`，例如 Zerevia、Nookcraft。
- 主面板标签：关键词任务、站点配置、运行逻辑、模型设置、新增站点。
- API Key 输入框：运行时传给生成接口，不写入项目文件。
- 导入数据文件/ZIP：进入通用导入流程。
- 导出 CSV：下载当前站点关键词表。
- 下载文章 ZIP：打包下载当前站点 `outputs/`。
- 元数据表格：下载 `meta-table.xlsx`。

切换站点时，前端会重新加载关键词、配置、模板，并清空已选状态，关闭抽屉。

## 4. 左侧运行控制区

左栏固定宽度 280px，所有面板共用。

### 4.1 统计数字

四个统计块：

- 总计：关键词任务总数。
- 已完成：状态为 `done` 的文章数量。
- 待生成：未生成或排队中的任务数量。
- 失败：状态为 `failed` 的任务数量。

这些数字来自 `/api/sites/:siteId/keywords` 返回的关键词行状态。

### 4.2 运行控制

按钮包括：

- 运行已选文章生成：只对当前勾选的关键词执行文章生成。它会调用 `generate-article`，要求该关键词已经有大纲；如果要一键全量生成，先点击表头复选框全选。
- 运行已选大纲：只对当前勾选的关键词批量生成大纲，用于先审大纲再生成正文。
- 重试失败：重新执行失败任务。
- 重置全部：把全部任务状态改回待生成。
- 重置已选：只重置勾选行。
- 并发数滑块：控制前端批量请求并发数。

当前文章生产逻辑是明确两阶段：先生成大纲，再根据大纲生成文章。单行操作中可以分别执行「纲」和「生成」；批量操作以“已选”为核心，通过表头全选保留一键全量处理能力。这样可以避免误触全站生成，也更适合先抽查大纲、再批量生成正文的运营流程。

### 4.3 运行日志

运行日志位于左侧底部，最新日志在最上方。用途：

- 展示导入结果。
- 展示保存配置成功或失败。
- 展示大纲生成、文章生成状态。
- 展示 API 错误、QA 失败、文件下载等反馈。

产品原则：普通 API 错误不弹窗，而是进入日志，避免打断运营流程。

## 5. 面板一：关键词任务

关键词任务是日常生产内容的主工作台。

### 5.1 关键词表格

表格字段：

- 复选框：用于批量重置或未来批量操作。
- 关键词：核心关键词，可内联编辑。
- URL Slug：文章输出文件名和 URL slug，可内联编辑。
- 优先级：P1/P2/P3。
- 文章类型：购买指南、价格指南、对比评测、FAQ 问答、操作教程、支柱文章等。
- 目标字数：用于 Prompt 和 QA 参考。
- 状态：待生成、生成中、大纲完成、已完成、失败。
- QA 分：文章生成后的综合评分。
- 操作：预览、生成大纲、生成文章、删除。

关键词表来自 `sites/[siteId]/keywords.csv`，由 `engine/keyword-store.js` 读写。

### 5.2 新增关键词

点击「新增关键词」展开表单。

字段包括：

- 关键词。
- URL Slug，留空可自动生成。
- 优先级。
- 文章类型。
- 目标字数。
- 搜索意图。
- 次要关键词。
- 变体/长尾词。
- 内链 URL。
- 文章方向/备注。

保存后调用 `POST /api/sites/:siteId/keywords`，写入 `keywords.csv` 并进入待生成状态。

### 5.3 文章抽屉

点击预览按钮后，从右侧滑出抽屉。抽屉是生产结果的集中查看区。

抽屉标签页：

- 大纲预览：读取 `outputs/[slug].outline.json` 或 `outlines/[slug].json`。
- 文章预览：渲染 `outputs/[slug].html`，并结合 `style-reference.json` 做基础样式预览。
- HTML 源码：显示原始 HTML。
- QA 检查：读取 QA 结果，显示综合得分、硬性拦截、警告和各项评分。
- 发布包：展示 SEO 标题、Meta Description、ALT 文案、Schema JSON-LD、发布前检查清单。
- 数据包：展示大纲、QA、发布包、生成 Prompt、关键词行等结构化数据。

下载单篇文章时，应使用 `/api/sites/:siteId/articles/:slug?download=1`，避免浏览器直接打开 HTML 源码页面。

## 6. 面板二：站点配置

站点配置面板负责管理某个站点的业务数据和品牌规则。顶部显示当前站点，并提供「保存所有更改」和「删除当前站点」。删除站点有二次确认，避免误删。

### 6.1 品牌设置：site.json

对应文件：`sites/[siteId]/site.json`

字段包括：

- 网站 ID。
- 网站名称。
- 域名。
- 语言。
- 网站定位。
- 目标受众。
- 品牌角色。
- 转化目标。
- 写作语气。
- 禁止规避的写作风格。
- 必须表达。
- 禁止表达。

后端读取后会进入 `ctx.site`，被 Prompt 构建和 QA 规则使用。

### 6.2 作者档案：author.json

对应文件：`sites/[siteId]/author.json`

字段包括：

- 作者姓名。
- 职位/身份。
- 背景介绍。
- 写作风格说明。
- 故事素材库。

故事素材库支持 JSON 数组结构，适合存放 Claude Project 中的一手经验素材、编号故事、真实客户问题、创始人经历等。此前发现的 `[object Object]` 显示问题已经修复：UI 现在会保留 JSON 结构，不再把对象数组错误转成字符串。

生成正文时，故事素材库会被作为“已批准的一手经验来源”传入 Prompt。模型应从故事库中选择与当前主题最相关的细节，改写成可直接发布的自然段落，而不是输出【需要一手经验】、TODO、E-E-A-T marker 等编辑占位符。如果故事库里没有完全匹配的素材，模型应写成更克制的事实/观察表达，不能留下待补标记。

### 6.3 品类知识：knowledge.json

对应文件：`sites/[siteId]/knowledge.json`

核心区块：

- 术语表：中英文术语和释义。
- 权威事实：可验证事实、行业标准、材料参数。
- 买家高频问题：FAQ 和回答。

这些数据用于 Prompt 中的事实约束，防止模型凭空发挥。

### 6.4 内链库：links.json

对应文件：`sites/[siteId]/links.json`

核心区块：

- 专题页：最高优先级。
- 已发布博客：用于同主题互链和关键词蚕食判断。
- 产品分类页：用于转化导向 CTA 和产品线支撑。

`engine/link-matcher.js` 会根据关键词和文章类型匹配合适内链。

### 6.5 样式参考：style-reference.json

对应文件：`sites/[siteId]/style-reference.json`

用途：保存目标网站文章 HTML 的排版和组件风格参考。

当前 UI 已改为一个大输入框 `styleBrief`，运营人员可以直接用自然语言填写样式规则，而不是拆成很多字段。结构化旧字段仍会保留，但生成时优先读取 `styleBrief`。

建议写清楚：

- 文章根节点 class，例如 `<article class="zerevia-article">`。
- 标题、段落、列表、表格的阅读节奏。
- FAQ 是否使用 `<details><summary>...</summary></details>` 手风琴。
- 引用块、经验块、提示块、警告块、CTA、Related links 的 HTML 结构和 class。
- 颜色和字体气质，例如 deep purple、lavender accent、warm ivory、editorial calm 等。
- 禁止输出 full-page HTML、`<style>`、`<script>`、`<head>`、`<body>`、nav、footer。

它不直接替代目标站主题 CSS，但会强约束模型输出带 class 的 CMS 可嵌入 HTML。最终视觉效果仍取决于目标站是否有对应 CSS；如果目标站没有这些 class 的样式，需要在主题或文章模板中补 CSS。

### 6.6 Project 指令：project-instructions.md

对应文件：`sites/[siteId]/project-instructions.md`

这是专门为 Claude Project 这类长规则设计的模块。建议存放：

- 你是谁。
- 品牌世界观。
- 写作禁区。
- 合规边界。
- E-E-A-T 规则。
- 文章结构规则。
- 发布检查清单。
- 标准输入/输出格式。

优先级建议：

1. 结构化配置文件优先，例如 site.json、author.json、knowledge.json。
2. Project 指令作为整体写作框架。
3. 单篇关键词任务中的备注和方向作为最具体约束。

如果三者冲突，越具体、越接近当前文章任务的信息优先。

## 7. 面板三：运行逻辑

运行逻辑面板用于调整生成策略和 QA 标准。

### 7.1 文章类型策略：article-types.json

用途：定义不同文章类型的生成边界，但不保存固定标题句式或固定开头模板。

当前 UI 已改为单个 JSON 填写框，而不是拆成多张字段卡片。这样做的原因是：文章类型策略本身应该服从全局 Project 指令和单篇关键词任务，如果在 UI 中硬编码 `titleFormat` 或 `openingTemplates`，会导致文章套模板、同质化，并且容易与站点 SOP 冲突。

推荐保留字段：

- 类型 ID：例如 `buying-guide`、`faq`、`how-to`。
- `wordRange`：默认字数范围，关键词行的 `targetwordcount` 优先级更高。
- `schemaType`：默认 Schema 类型，例如 `Article`、`FAQPage`、`HowTo`。
- `strategy`：文章类型原则，用来描述读者意图、结构重点、证据要求和转化角色。
- `structureNotes` / `avoid`：可选，用来说明结构注意事项和应避免的写法。

不建议保留字段：

- `titleFormat`：容易让标题同质化。
- `openingTemplates`：容易让开头句重复，尤其会和 Project 指令中的品牌写法冲突。

当前前端保存时会自动移除 `titleFormat` 和 `openingTemplates`。全局 `templates/article-types.json` 也已改为通用 fallback 策略，不包含 Nookcraft、Zerevia 或其它具体站点的硬编码句式。站点差异应放在 `sites/[siteId]/templates/article-types.json`、`project-instructions.md`、知识库和关键词任务中。

### 7.2 Prompt 区块：prompt-sections.json

用途：维护通用 Prompt 片段。

典型区块：

- 作者声明格式。
- 观察性结论写法。
- 内链规则说明。
- CTA 风格要求。

这些区块会被 `prompt-builder.js` 组合进大纲 Prompt 和文章 Prompt。

### 7.3 QA 规则：qa-rules.json

用途：定义硬性拦截、警告和评分权重。

QA 分层：

- Hard Fail：必须拦截，例如出现禁用词、缺少作者模块、没有核心关键词等。
- Warning：需要人工复核，例如没有产品页内链、存在关键词蚕食风险、缺少一手经验等。
- Score Weights：SEO 结构、品牌匹配、内容有用性、可读性、发布就绪度。

当前 QA 已支持硬性拦截和警告分开显示。分数不是唯一判断标准：84 分也可能因为 Hard Fail 而不可发布。

## 10. 通用导入数据文件/ZIP

导入入口是当前版本的重要模块。它解决的问题是：实际站点数据通常不止一个 CSV，而是一整包配置文件、模板、报告和关键词。

### 10.1 前端流程

1. 用户点击「导入数据文件/ZIP」。
2. 选择多个文件，或直接选择 `.zip`。
3. 前端读取文件内容，`.xlsx` 和 `.zip` 使用 base64。
4. 调用 `/api/sites/:siteId/import-preview`。
5. 后端解析 ZIP，识别每个文件的目标位置和导入状态。
6. 前端展示待导入文件列表。
7. 用户勾选需要导入的文件。
8. 调用 `/api/sites/:siteId/import-files` 正式写入。

### 10.2 可识别文件

核心配置：

- `site.json` → `sites/[siteId]/site.json`
- `author.json` → `sites/[siteId]/author.json`
- `knowledge.json` → `sites/[siteId]/knowledge.json`
- `links.json` → `sites/[siteId]/links.json`
- `style-reference.json` → `sites/[siteId]/style-reference.json`
- `project-instructions.md` → `sites/[siteId]/project-instructions.md`

关键词：

- `keywords.csv` → 合并导入 `keywords.csv`
- `.xlsx` 关键词表 → 转成 CSV 后合并导入

模板：

- `article-types.json` → `sites/[siteId]/templates/article-types.json`
- `prompt-sections.json` → `sites/[siteId]/templates/prompt-sections.json`
- `qa-rules.json` → `sites/[siteId]/templates/qa-rules.json`

生成资料：

- `outline.json` 或 `*.outline.json` → `outlines/[slug].json`
- `data-pack.json` 或 `*data-pack.json` → `outputs/[slug].data-pack.json`

报告和参考文件：

- `reports/*.csv`
- `reports/*.json`
- `reports/*.md`
- `migration_notes.md`
- `source_manifest.json`
- `url_inventory.csv`
- `keyword_filter_report.csv`

这些会进入 `sites/[siteId]/docs/`。

### 10.3 重复识别

导入预览会标记：

- 新文件：目标不存在，可导入。
- 覆盖：目标存在但内容不同。
- 重复跳过：目标存在且内容相同。
- 未识别：没有匹配导入规则。
- 错误：JSON 无法解析、文件内容异常等。

关键词导入会按 slug 或 keyword 去重，避免重复写入同一批关键词。

## 11. 后端 API 说明

### 11.1 站点管理

## 12. 执行链路：从关键词到文章

### 12.1 加载上下文

`config-loader.js` 读取站点目录，组装上下文 `ctx`：

- `ctx.site`
- `ctx.author`
- `ctx.knowledge`
- `ctx.linkIndex`
- `ctx.styleReference`
- `ctx.projectInstructions`
- `ctx.templates`
- `ctx.outputDir`

### 12.2 构建大纲 Prompt

`prompt-builder.js` 使用：

- 当前关键词行。
- 文章类型策略。
- 品牌设置。
- 作者档案。
- 知识库。
- 内链库。
- Project 指令。
- Prompt 区块。

输出大纲生成所需的 systemPrompt 和 userPrompt。

### 12.3 调用模型生成大纲

`api.js` 中的 `generateOutline()` 调用模型接口。模型必须返回可解析 JSON。后端会做 normalize，确保至少有：

- slug
- keyword
- articleType
- workingTitle
- sections
- faq
- qaRisks

大纲写入：

- `outputs/[slug].outline.json`
- `outlines/[slug].json`

### 12.4 基于大纲生成文章

`generateArticleFromOutline()` 会读取大纲，再调用 `generator.js` 生成文章 HTML。

输出结果包括：

- 原始生成内容。
- 清洗后的 HTML。
- SEO meta。
- QA 结果。
- data-pack。

写入文件：

- `outputs/[slug].html`
- `outputs/[slug].qa.json`
- `outputs/[slug].data-pack.json`
- `outputs/[slug]-data-pack.json`
- `outputs/meta-table.xlsx`

### 12.5 QA 检查

`qa.js` 对清洗后的 HTML、关键词行和上下文执行检查。

输出：

- 综合得分。
- 分项得分。
- hardFails。
- warnings。
- status：pass / review / blocked。

重要逻辑：

- 有 hardFail 时，不能仅因为分数高就视为通过。
- warning 不一定阻止发布，但应进入人工复核。
- 站点级 QA 规则可以覆盖通用规则。

### 12.6 元数据输出

`meta-writer.js` 写入 `meta-table.xlsx`，包括：

- Sheet1：文章记录。
- Sheet2：SEO 标题和 Meta Description。
- Sheet3：ALT 文案。
- Sheet4：Schema。

发布包接口会合并 HTML、meta-table 和 data-pack 中的信息，方便运营直接复制发布。

## 13. 数据优先级和冲突处理建议

当前建议的优先级如下：

1. 单篇关键词任务字段：最具体，例如 article direction、internal links、target word count。
2. 结构化站点配置：site.json、author.json、knowledge.json、links.json、style-reference.json。
3. 站点级 Project 指令：project-instructions.md。
4. 站点级模板：sites/[siteId]/templates/*.json。
5. 全局模板：templates/*.json。
6. 引擎内置默认值。

冲突示例：

- 如果 Project 指令说“每篇 1200-1800 词”，但关键词行写了 `targetwordcount=900`，应以关键词行为准。
- 如果 Project 指令中有内链规则，但 links.json 中没有对应专题页，应由 QA 给出 warning，而不是模型编造 URL。
- 如果中文 SOP 中写了英文输出要求，应该保留中文配置，但 Prompt 明确 `Write reader-facing content in English`。

## 14. 当前已知优化方向

### 14.0 新增站点资料向导

新增站点页现在分为两个阶段：

- 第一阶段填写基础信息：网站 ID、网站名、域名、语言、定位、作者、语气、受众和表达禁区，用于创建站点目录和默认文件骨架。
- 第二阶段上传网站变量数据：支持 `.json`、`.csv`、`.md`、`.txt`、`.xlsx` 和 `.zip`。ZIP 会在后端解压后逐个文件识别。

资料向导会展示必备文件清单：

- `site.json`：品牌/网站变量。
- `author.json`：作者档案和故事素材库。
- `knowledge.json`：品类知识库。
- `links.json`：内链库。
- `style-reference.json`：文章 HTML 样式参考。
- `project-instructions.md`：站点级 Project 指令。
- `keywords.csv`：关键词规划表。

后端新增接口：

- `POST /api/sites/:siteId/setup-preview`：接收上传文件，先用现有导入规则做启发式识别；如果请求中带有 API Key，则调用模型判断文件属于哪个模块、应写入哪个目标文件、置信度是多少，并返回可勾选的识别列表。
- `POST /api/sites/:siteId/setup-apply`：只写入用户勾选的识别结果。JSON 配置会与现有配置做递归合并，关键词表走去重导入，模板文件写入站点级 `templates/`，无关文件会被标记为 `ignore`。

这个设计和“站点配置”不冲突：资料向导负责把原始资料转成程序可读格式；站点配置页负责人工复核和二次编辑。和“导入数据文件/ZIP”也不冲突：资料向导偏建站初始化，通用导入偏已有站点的数据维护。

建议后续迭代按优先级推进：

1. 为批量任务增加更细的运行前检查：例如选中项是否已有大纲、是否缺少 API Key、是否缺少关键内链。
2. 样式参考进一步增强：从目标站 HTML 自动提取 typography、容器、按钮和文章模块 class。
3. QA 规则和 Prompt 规则统一命名，避免同一规则在多个地方重复表达。
4. Project 指令拆分为结构化规则：身份、合规、输出格式、E-E-A-T、内链、Schema。
5. 增加“生成前检查”：缺少专题页、缺少产品页、缺少故事素材时先提示。
6. 继续优化故事素材匹配：根据关键词主题自动选择更精准的 storyBank 条目，并在数据包中记录引用来源。
7. 单篇下载按钮统一使用 attachment 下载，避免打开源码页。
8. 为目标站补充或同步对应 CSS：styleBrief 会让模型输出 class 和组件结构，但最终美化仍需要网站主题 CSS 支持。
9. 导入列表增加“已导入历史”，让运营看到某次 ZIP 写入了哪些文件。
10. GitHub 提交流程固化：程序代码提交，真实站点数据默认不提交。

## 15. 给老板汇报时可以这样概括

这套系统的价值不是“让 AI 写文章”，而是把过去散落在人工 SOP、Claude Project、关键词表、知识库、内链表和发布检查清单里的内容生产流程，沉淀成一个可重复运行的后台。

它带来的效率提升主要体现在：

- 站点复制：新站点只需要导入配置包和关键词表，就能复用生产框架。
- 标准一致：品牌语气、禁用词、合规规则、内链规则和 QA 标准集中管理。
- 过程可见：从关键词、大纲、文章、QA 到发布包，每一步都有界面和文件记录。
- 质量可控：不是直接让模型输出成品，而是先大纲、再正文、再 QA、再发布包。
- 人机协作：中文母语运营可以维护中文配置，模型按要求输出英文内容。
- 可持续迭代：每次真实生产中的问题都可以沉淀回模板、Project 指令或 QA 规则。

最终目标是从“人工逐篇写博客”升级为“运营维护内容系统，系统批量生产可审核文章”的模式。
